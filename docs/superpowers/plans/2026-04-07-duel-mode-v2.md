# Duel Mode V2 Plan

## Goal

Implement a production-grade `duel mode` for the global `gemini-consult` stack so Codex and Gemini can produce competing solutions from the same brief, run deterministic machine judging first, and let Codex issue the final verdict. The result must be fast enough for real work, Windows-safe, resumable, and robust under long-running Gemini passes.

Implementation status on 2026-04-07:

- completed locally in `C:\Users\yehor\Desktop\codex-gemini-consult`
- verified with parser checks and local smoke tests

## Non-Goals

- Do not build a purely LLM-only judge.
- Do not merge diffs automatically without an explicit integration pass.
- Do not couple the first implementation to one repository's build/test stack.
- Do not depend on Unix-only shell behavior.
- Do not replace the existing `gemini-consult` modes; extend them.

## Architecture Decision

Build duel mode as a first-class extension of the skill pack with a dedicated orchestration layer, not as an ad-hoc prompt pattern.

### Core Roles

1. `Codex candidate`
   - Codex creates its own plan and patch in an isolated worktree.

2. `Gemini candidate`
   - Gemini creates its own plan and patch from the same shared brief and context packet.

3. `Machine judge`
   - Runs deterministic checks and computes a scoreboard.

4. `Codex final judge`
   - Reviews both candidates plus machine evidence and returns one of:
   - `accept-codex`
   - `accept-gemini`
   - `merge-best-of-both`
   - `reject-both`

### Why This Shape

- Deterministic checks should decide everything they can.
- Gemini should not be the sole judge of a patch it may have authored.
- Codex already owns integration and verification by global policy.
- A machine-first scoreboard keeps the process fast and reduces subjective noise.

## Shared Brief Contract

Every duel run must generate one normalized shared brief before either candidate starts.

Required sections:

- Goal
- Deliverable
- Locked Scope
- Allowed Change Surface
- Forbidden Change Surface
- Relevant Files
- Validation Commands
- UX/Design Constraints
- Contract Constraints
- Risks

Rules:

- Both candidates must receive the exact same brief.
- Both candidates must receive the same root path.
- Both candidates must receive the same context manifest.
- If scope is locked, the brief must say so explicitly.

## Candidate Isolation

Use git worktrees whenever the project is a git repository.

Default branch/worktree naming:

- `codex/duel-<slug>-codex`
- `codex/duel-<slug>-gemini`

Fallback when project is not a git repository:

- mirror the repo into two temp working copies under the duel artifact root
- mark the run as `no-git-fallback`

Isolation rules:

- no shared write tree between candidates
- no patch application into the source tree during candidate generation
- final integration only after verdict

## Machine Judge V1 Rubric

The machine judge runs first and emits a weighted scoreboard.

### Deterministic Dimensions

- `tests_pass`
- `lint_pass`
- `typecheck_pass`
- `build_pass`
- `runtime_smoke_pass`
- `locked_scope_pass`
- `forbidden_files_touched`
- `changed_file_count`
- `diff_line_count`
- `critical_surface_delta`
- `contract_check_pass`
- `a11y_check_pass`
- `visual_smoke_pass`

### Initial Weighting

- verification gates are binary gates first, score second
- any failing required gate blocks automatic acceptance
- among passing candidates, prefer:
  - fewer forbidden or risky touches
  - smaller safe diff
  - fewer changed files when outcomes are equivalent
  - stronger visual/runtime evidence for UI tasks

### Verdict Logic

- if one candidate passes required gates and the other fails, the passing candidate wins unless Codex sees a severe hidden risk
- if both fail required gates, verdict defaults to `reject-both` or targeted salvage
- if both pass, Codex compares engineering quality and may choose `merge-best-of-both`

## Codex Final Judge Rubric

Codex final judgment should consider only dimensions the machine cannot settle well:

- behavior preservation confidence
- maintainability
- architecture fit to repo patterns
- integration cost
- UX quality and coherence
- likely hidden regressions
- whether one candidate solved the right problem more faithfully

Codex must not overrule a hard failing deterministic gate without an explicit reason in `judge/verdict.md`.

## Fast Path Design

Because speed matters, duel mode needs a staged fast path.

### Fast Path A

Use full duel only when one of these is true:

- architecture change
- refactor with behavior preservation
- flaky bug
- broad UI redesign
- high-value risky task selected by user

### Fast Path B

For medium tasks:

- generate the shared brief once
- let Codex prepare its candidate locally
- ask Gemini only for its candidate plan and implementation package
- run machine judge on both

### Fast Path C

For trivial tasks:

- skip duel mode entirely
- fall back to normal `gemini-consult`

## Screenshot Problem To Solve Explicitly

Observed class of failure:

- broad Gemini runs on Windows appear "stuck" when they are actually long-running
- external shell timeout kills the run before the long pass completes
- multiline prompt transport through shell quoting or pipeline glue is fragile

Planned fix:

1. split `task duration budget` from `transport timeout budget`
2. duel orchestrator must assign long timeout defaults for Gemini candidate generation
3. every long Gemini run must use:
   - `stream-json`
   - prompt file or direct stdin payload
   - no giant quoted shell strings
4. process supervisor must log heartbeat timestamps from stream events
5. if there is no event for a configured idle threshold, emit `suspected-stall` instead of killing immediately
6. only kill the process after a second, larger hard timeout is exceeded

This specifically addresses the behavior shown in the screenshot: the orchestration layer treated a long Gemini pass as transport-level slowness instead of a valid long-running agent task.

## Windows Failure Modes And Predicted Fixes

### 1. Encoding drift across PowerShell, cmd, Node, and Gemini

Risk:

- broken Unicode
- malformed JSONL / `stream-json`
- corrupted prompts

Mitigation:

- use UTF-8 without BOM for prompt files and JSON artifacts
- set explicit stdio encodings in all spawned processes
- avoid `cmd /c type file | ...` as a transport primitive

### 2. Shell quoting failures for large multiline prompts

Risk:

- PowerShell parser errors
- truncated prompt payloads
- accidental interpolation of `$` and backticks

Mitigation:

- prefer `-PromptFile` and file-backed payloads for any prompt above a small threshold
- forbid giant inline command strings in duel orchestration

### 3. Path issues with spaces, non-ASCII, or cross-drive worktrees

Risk:

- wrong cwd
- broken file reads
- bad relative path assumptions

Mitigation:

- normalize all paths to absolute paths early
- use `-LiteralPath` for PowerShell file operations
- store a resolved path manifest in `context-manifest.json`
- verify the working directory exists before candidate launch

### 4. MAX_PATH and deep artifact trees

Risk:

- failed file creation in nested duel artifacts or worktrees

Mitigation:

- keep duel IDs short
- flatten artifact filenames
- avoid deeply nested temp paths
- add a path-length preflight warning

### 5. Process orphaning and zombie Gemini/Node children

Risk:

- stale background processes
- locked files
- false busy ports

Mitigation:

- maintain a tracked process table in `resume.json`
- on cancellation or failure, kill the full process tree
- add cleanup on next launch for stale duel processes

### 6. stdout/stderr deadlocks or blocked long-running output

Risk:

- parent process appears hung while child is blocked on full buffer

Mitigation:

- always drain stdout and stderr concurrently in the orchestrator
- keep stream-json reading line-based and non-blocking where possible

### 7. Antivirus or file indexing interference in temp locations

Risk:

- prompt files appear missing or locked
- patch files fail to write atomically

Mitigation:

- write duel artifacts under the project-local `.codex/duels` root instead of random system temp when possible
- add retry-on-open for artifact writes

### 8. Git worktree collisions and branch residue on Windows

Risk:

- stale worktrees
- branch already exists
- cleanup failures due to file locks

Mitigation:

- deterministic branch naming with collision suffixes
- preflight cleanup and reuse logic
- delayed cleanup mode when files are locked

### 9. CRLF vs LF diff noise

Risk:

- fake diff inflation
- unfair machine scoring

Mitigation:

- normalize score metrics on parsed diff hunks, not raw file byte counts
- honor repo `.gitattributes` if present

### 10. Missing toolchain commands in candidate environments

Risk:

- one candidate fails due to environment mismatch rather than code quality

Mitigation:

- run a shared environment preflight once before duel start
- mark gates as `blocked-environment` instead of `candidate-fail` when the environment is broken for both

### 11. Non-git project roots

Risk:

- duel mode unusable outside repos

Mitigation:

- support mirrored working copies under `.codex/duels/<id>/workspace-*`
- clearly flag reduced confidence and reduced cleanup guarantees

### 12. Screenshot/UI verification failures on headless Windows stacks

Risk:

- UI candidate judged without actual surface evidence

Mitigation:

- detect available verification stack first
- prefer browser-native screenshots when available
- fall back to DOM/runtime smoke if full screenshot path is unavailable
- never claim visual superiority without evidence
