# Duel Mode V3 Plan

## Goal

Upgrade `gemini-consult` and `gemini-duel` from a working v2 orchestration layer into a production-grade collaborative system that:

- prevents oversized Gemini runs before they stall
- narrows scope automatically instead of forcing Codex to rescue prompts manually
- improves Codex/Gemini cooperation as a staged workflow rather than a single giant generation call
- stays fast and deterministic on Windows
- produces reusable artifacts, resumable state, and predictable verdicts for risky tasks

V3 is not about making Gemini "wait longer". It is about making the system ask Gemini for the right-sized task in the right stage, with the right transport and the right fallback rules.

## Why V3 Exists

V2 solved the basic orchestration problem:

- shared brief
- isolated candidates
- Gemini candidate package path
- machine judge
- verdict artifact

But it still leaves too much burden on Codex during hard tasks:

- Codex has to notice when the Gemini packet is too large
- Codex has to manually narrow broad redesign prompts into route-scoped or contract-scoped prompts
- long Gemini runs can still be valid but operationally wasteful
- the system treats “one huge candidate generation pass” as the default when many tasks should be staged

Observed failure pattern:

- Codex sends Gemini too much context
- Gemini shifts from “implementation package” into “heavy agent session”
- Codex waits, then manually compacts the task
- the eventual winning path is usually a narrower packet that the launcher should have chosen automatically

Additional observed failure pattern:

- Gemini can produce the right direction or usable code, but the answer is too large or too stream-heavy to trust the interactive shell buffer alone
- Codex then manually tries to “save the rest into a file” so the code can be recovered without truncation
- this is the wrong abstraction boundary: the launcher should persist the full raw Gemini output automatically and parse from artifacts instead of depending on terminal rendering

V3 fixes that class of failure structurally.

## Non-Goals

- Do not turn duel mode into a fully autonomous multi-hour planner by default.
- Do not let Gemini choose arbitrary repo scope on its own.
- Do not remove deterministic judging in favor of taste-based judging.
- Do not replace `gemini-consult` one-shot modes for normal work.
- Do not make Unix-only assumptions.
- Do not make the orchestration dependent on one repository’s stack.

## Current State

V2 already provides:

- working `gemini-consult.ps1`
- working `gemini-duel.ps1`
- git and non-git candidate isolation
- machine scoreboard
- verdict artifact
- Windows-safe fallbacks for PowerShell compatibility issues already encountered

Main gaps that V3 must close:

- no hard context budget enforcement before Gemini candidate generation
- no automatic prompt packetization
- no stage-based candidate pipeline
- no auto-reroute when Gemini packets are too broad or stall
- no structured “scope audit” phase before implementation
- no route-scoped / file-scoped task decomposition for broad redesigns
- no explicit collaboration metrics telling Codex when to use normal consult vs staged duel

## Architecture Decision

V3 will keep the v2 artifact-and-scoreboard architecture, but change the candidate generation model from:

- one shared brief
- one big Gemini candidate generation call

to:

- one shared duel ledger
- staged decision pipeline
- packetized prompts
- automatic narrowing and rerouting

### Core Principle

Broad tasks should not be sent to Gemini as one giant implementation request unless the system has positive evidence that the packet is small enough and the scope is stable enough.

### New Core Workflow

1. `preflight`
2. `scope-audit`
3. `brief-compact`
4. `candidate-plan`
5. `candidate-package`
6. `machine-judge`
7. `final-verdict`
8. optional `merge-best-of-both`

This becomes the default duel pipeline for broad or risky tasks.

## V3 Design Themes

### 1. Packetization Instead Of Giant Prompt Strings

Gemini should no longer receive one monolithic prompt assembled from a huge here-string when the task is broad.

Instead, V3 builds a packet directory under the duel artifact root:

- `packet/objective.md`
- `packet/constraints.md`
- `packet/scope.json`
- `packet/context-manifest.json`
- `packet/relevant-files/<n>.txt`
- `packet/output-contract.md`
- `packet/stage.md`

Gemini receives:

- a short controller prompt
- the explicit project root
- the stage name
- the packet file paths

Benefits:

- less shell quoting fragility on Windows
- reproducible prompts
- easier retries, resume, and debugging
- better visibility into why a packet was considered “too large”

### 2. Hard Context Budget Before Candidate Generation

V3 introduces explicit budgets before Gemini candidate generation.

Metrics:

- `context_file_count`
- `total_context_chars`
- `largest_file_chars`
- `prompt_chars`
- `packet_chars`
- `validation_command_count`

Default decision classes:

- `small`
  - direct candidate generation allowed
- `medium`
  - compact brief required first
- `large`
  - stage split required
- `oversized`
  - generation blocked until narrowed

The launcher must never silently send an oversized packet to Gemini.

If a packet is oversized:

- it does not proceed to full candidate generation
- it writes a machine-readable reason into the duel ledger
- it reroutes automatically into a narrower stage

### 3. Stage-Based Duel Pipeline

Broad tasks should be decomposed automatically.

New stages:

#### Stage A. `scope-audit`

Goal:

- determine what is allowed to change
- determine what must remain untouched
- identify the minimal file surface
- classify task shape

Outputs:

- `scope-audit.md`
- `task-shape.json`

Task shape classes:

- `single-file-fix`
- `route-scoped-implementation`
- `shared-shell-redesign`
- `primitives-and-layout`
- `docs-only`
- `architecture-only`
- `broad-ui-redesign`
- `behavior-preserving-refactor`

#### Stage B. `brief-compact`

Goal:

- compress the task into a normalized, stage-safe brief

Outputs:

- `compact-brief.md`

Rules:

- include only the minimum necessary relevant files
- include locked scope explicitly
- include allowed and forbidden surfaces
- include expected output schema

#### Stage C. `candidate-plan`

Goal:

- ask Gemini for the narrowest execution plan before asking for code

Outputs:

- `gemini/plan.md`
- optional `codex/plan.md`

Rules:

- no code generation yet for large tasks
- the plan should confirm file surface and preservation rules

#### Stage D. `candidate-package`

Goal:

- request the actual implementation package only after scope and file surface are stable

Outputs:

- `gemini/summary.md`
- `gemini/diff.patch`
- `gemini/candidate.json`

#### Stage E. `judge`

Same machine-first logic as v2, but with richer evidence and better blocked-environment classification.

### 4. Auto-Reroute On Stall Or Oversize

This is the “fix it once and for all” layer.

If any Gemini candidate run hits one of these states:

- packet budget exceeded
- idle heartbeat threshold exceeded
- hard timeout risk is approaching
- stage/task shape mismatch detected
- Gemini starts producing broad redesign framing when route-scoped output was requested

then the system must automatically reroute rather than waiting for Codex to manually shrink the task.

Auto-reroute actions:

- `broad-ui-redesign` -> `shared-shell-redesign`
- `shared-shell-redesign` -> `route-scoped-implementation`
- `many-files` -> `top-N-file packet`
- `full-context` -> `compact brief + critical files only`
- `heavy implementation pass` -> `candidate-plan` first

Every reroute must be recorded in:

- `reroute-log.json`
- `resume.json`

### 5. Better Codex/Gemini Cooperation

V3 should improve collaboration, not only constrain Gemini.

Codex responsibilities:

- project inspection
- selecting duel vs normal consult
- maintaining source-of-truth constraints
- integrating the accepted result
- final engineering judgment

Gemini responsibilities:

- high-quality candidate plan
- high-quality candidate package
- structured critique when requested
- not widening task scope

New collaboration rules:

- Gemini is not asked to “own the task”; it is asked to own one stage or one candidate
- Codex should not manually compact prompts when the orchestrator can do it
- Codex should receive explicit machine reasons when Gemini was rerouted or blocked

### 6. Full-Response Capture Without Terminal Truncation

This is the “fix it properly” layer for the screenshot class where Codex has to manually save the rest of Gemini’s answer into a file.

V3 rule:

- Gemini output must be persisted automatically to artifacts before Codex tries to read or parse it
- package extraction must operate on the persisted artifact, not on whatever happened to render in the shell buffer
- terminal output becomes a progress surface, not the source of truth

New artifact files per Gemini stage attempt:

- `gemini/attempt-<n>-raw.txt`
- `gemini/attempt-<n>-normalized.txt`
- `gemini/attempt-<n>-package.json` when package extraction succeeds
- `gemini/attempt-<n>-metadata.json`

Rules:

- always tee long Gemini stdout into a raw artifact file
- always store stderr separately
- if `stream-json` is used, also persist the reconstructed assistant text
- if marker-based package extraction is used, persist both:
  - raw response
  - extracted JSON body
- Codex should never need to manually rerun a command “just to get the full answer into a file”

## New Decision Engine

### Duel Entry Decision

Before entering duel mode, the system classifies the task:

- `normal-consult`
- `fast-duel`
- `staged-duel`

Rules:

- trivial or narrow task -> `normal-consult`
- risky but contained task -> `fast-duel`
- broad UI redesign, architecture, refactor, or repeated Gemini overload history -> `staged-duel`

### Packet Budget Decision

Budget outcome:

- `allow`
- `compact-first`
- `split-stage`
- `block-until-narrowed`

### Stall Decision

During Gemini runs:

- `healthy`
- `suspected-stall`
- `reroute-required`
- `hard-timeout-failure`

## Windows Hardening V3

V3 must explicitly handle these Windows-specific issues.

### 1. Long Runs That Are Slow But Healthy

Fix:

- heartbeat timestamps from `stream-json`
- separate `idle timeout` from `hard timeout`
- mark `suspected-stall` before kill
- only reroute or kill after policy threshold

### 2. Heavy Prompt Assembly Via Here-Strings

Fix:

- packet files only for medium/large tasks
- no huge inline PowerShell strings in duel orchestration

### 3. PowerShell 5 Compatibility

Fix:

- keep compatibility helpers for relative paths and process arguments
- test under Windows PowerShell, not only `pwsh`

### 4. Locked Files In Worktrees Or Mirror Copies

Fix:

- preflight worktree reuse
- delayed cleanup mode
- process table in ledger

### 5. Unicode Path And Prompt Stability

Fix:

- all packet and JSON files written as UTF-8 without BOM
- explicit stdio encodings
- smoke tests with Unicode file names and Unicode content

## Metrics To Track

V3 should track operational quality, not just correctness.

- `% of Gemini candidate runs that required reroute`
- `% of oversized packets blocked before generation`
- average candidate package file count
- average packet size per task shape
- average time from duel start to verdict
- % of duels where machine recommendation and Codex final verdict match

These metrics should be stored in ledger-friendly JSON, even if they are not yet exposed in the UI.

## Likely Risks

- too aggressive rerouting may starve Gemini of necessary context
- task-shape classification may initially misclassify hybrid tasks
- route-scoped specialization may need repo-specific overrides later
- optional UI evidence hooks may be flaky on some Windows stacks
- raw response artifacts may grow large if retention is unbounded

Mitigations:

- log every reroute reason
- make thresholds configurable
- keep stage artifacts human-readable
- keep a manual override path for Codex
- add retention limits and optional artifact pruning policies
